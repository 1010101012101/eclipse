Platform/Core Component Planning

Current Work

Work will now be focused on Eclipse 3.0. Rather than rolling out version 2.2 which would contain fixes and minor feature requests, it has been decided that we will take on more challenging problems. Please read Why Eclipse "3.0"? and the Eclipse 3.0 Project Plan.

Being done concurrently with the 3.0 work are bug fixes for service releases for Eclipse 2.1. For a list of problems which will be fixed for the Eclipse 2.1.1 release, check out the Bugzilla bug reports by clicking here.

 

Major work items

Concurrency/Background Operations - One of the biggest complaints about Eclipse is that almost every user action locks the system for the duration of the operation. When you save a Java file you have to wait for the compiler to finish before you do anything, when you load a project from CVS you have to wait until it completes before you can edit code, etc. The limitations of the current model are increasingly evident in the context of the Rich Client Platform (RCP).

Dynamic Plug-ins - The current Eclipse plug-in activation story is "somewhat less than optimal". Plug-ins are automatically activated upon first reference and once a plug-in is started, it remains in that state for the duration of the Eclipse session. We would like to migrate to a model which would allow for plug-ins to be activated and de-activated within the same session. As well, when a user installs a new plug-in to their configuration, they must restart Eclipse in order for their changes to take effect. As part of the dynamic plug-in work, we are also investigating incremental registry resolution which means that users won't require a restart for new plug-in changes to take effect.

Team/Resource Integration - There currently exists a few hooks in the core code which allow Team providers the opportunity to perform operations on resources before/instead of the core default. We wish to allow for a pluggable mechanism which provides more control over these operations. Check out the document link for a good description of this problem on the Platform/Team web page.

Improve User Experience - As a first-time Eclipse user there are many non-obvious problems that you can run across and this plan item is being used to collect them all. Some current problems include (but are not limited to): default location of the workspace, separation of user settings and plug-in metadata, multiple workspaces, and translation of messages presented to the user from the Eclipse executable.

Remove Xerces Dependancy - The Eclipse SDK is shipped with a Xerces implementation which is approximately 3M in size. (source and binaries) This gets interesting as we move towards supporting a Rich Client Platform (RCP) and have applications which are interested in using minimal installs of Eclipse and perhaps even running Eclipse headless. In these cases 3M is a large percentage of the Eclipse base that they are using. In an effort to reduce the size of the standard Eclipse components required and to increase performance (speed) and size (memory), we are investigating removing our dependancy on the Xerces plug-in and instead including a light-weight, fast XML parser such as one based on the XMLPull technology.

Improve File Encoding Support - Eclipse 2.1 uses a single global file encoding setting for reading and writing files in the workspace. This is problematic; for example, when Java source files in the workspace use OS default file encoding while XML files in the workspace use UTF-8 file encoding. The Platform should support non-uniform file encodings.
