Performance in Eclipse
Performance is the most pressing issue facing Eclipse in the remaining 3.1 development period. Poor performance is a bug and should be tested for, tracked and fixed in the same way as other bugs. This site is a collection of resources and information aimed at helping plug-in developers do just that.
Performance Issues
Scalability is the issue. Its the buzzword in the halls. People are asking themselves,

What would happen if we had 10x more... ?

More plug-ins, more resources, more windows, menus, perspectives, more markers, ...

Scalability is essentially an amalgum of the classic speed/space tradeoff with a twist of complexity. A small, fast N^2 algorithm is great if N=10, but is not so good if N=100.

Where can you find improved scalability?

Plug-in startup time
There is more to startup time than just how long it takes the Eclipse workbench to come up. If your plug-in is slow to start, users have to wait for your function. This may happen at any point in a session. Plug-ins that start too early, needlessly activate others, load too much code and/or do too much work when first activated work against scalability.
Footprint
Footprint affects scalability as bloated, memory hungry applications tax the workstation's resources causing the system as a whole to slow down. Understand the space you are using. Do you need all that information? All the time? For all the elements? Is there a more efficient structure? Count the bytes!
Algorithms
Look for algorithmic improvements. Change the approach to the problem and you may get 2x, 5x or even 10x improvements. When N is small, speed is easy.
Context
Understand the whole picture. What does it mean to use some API provided by others. Some apparently simple functions may require megabytes of code/data to be loaded or seconds/minutes of computing to be done. Even if its not your API its still your problem if you are the one triggering it.
