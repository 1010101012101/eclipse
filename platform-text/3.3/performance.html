3.3 Performance Pass

Last revised 11:12 CET March 28, 2007

Plan

  1. investigate red performance tests and create bugs with 'performance' keyword where needed
  2. prioritize (see blow) bugs that have the performance keyword and look out for other performance related bugs
  3. prioritize (see blow) bugs reporting leaks
  4. start fixing the bugs according to assigned priority

Priorities

P1
  • performance test degradations
  • bugs that slow down the user in common scenarios, e.g. full build on startup
  • leaks that happen in common scenarios and grow the memory each time
P2
  • performance bugs that obviously slow down the user
  • all other leaks that grow each time the scenario is repeated
P3
  • other performance bugs that should be addressed for 3.3
  • leaks that are not real leaks but holding on to last instance
P4
  • not really an issue, e.g. corner cases and nice to have things
